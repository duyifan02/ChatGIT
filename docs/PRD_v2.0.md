# ChatGIT v2.0 产品需求文档 (PRD)

> 基于第一性原理的产品改进计划

---

## 一、产品现状分析

### 1.1 产品定位

ChatGIT 是一款浏览器扩展，为 ChatGPT 用户提供**对话分支的可视化、导航与恢复**功能。解决的核心问题是：ChatGPT 原生界面无法直观展示对话树的全貌，用户在多次重新生成或编辑消息后容易"迷路"。

### 1.2 当前能力 (v1.1.0)

| 能力 | 状态 |
|------|------|
| 对话树可视化 | 已实现 |
| 分支导航与跳转 | 已实现 |
| 回复版本检测与切换 | 已实现 |
| 全分支自动采集 | 已实现 |
| 本地持久化存储 | 已实现 |
| 明/暗主题自适应 | 已实现 |
| 调试日志面板 | 已实现 |
| Chrome + Firefox 支持 | 已实现 |

### 1.3 第一性原理拆解

回到用户的根本需求：

```
用户使用 ChatGPT → 产生多轮对话 → 对部分回答不满意 → 重新生成/编辑 → 产生分支
→ 核心需求：不丢失任何有价值的 AI 输出，并能高效地找到、对比、复用它们
```

当前产品只覆盖了**"找到"**这一环节。**"对比"和"复用"**是明确的未覆盖需求。

## 二、用户画像与场景分析

### 2.1 目标用户

| 用户类型 | 典型场景 | 当前痛点 |
|----------|----------|----------|
| 开发者 | 用 ChatGPT 调试代码，多次重新生成比较不同解法 | 无法并排比较两个分支的代码方案 |
| 写作者/文案 | 让 ChatGPT 生成多个版本的文案，挑选最佳 | 只能逐个切换查看，无法快速对比 |
| 研究者 | 深度对话探索一个主题，分支探索不同方向 | 无法标记重要发现，无法导出特定分支 |
| 日常用户 | 偶尔重新生成回答 | 需要一目了然的导航（当前已满足） |

### 2.2 使用频率分析

- **高频操作**：查看树 → 点击跳转（已优化）
- **中频操作**：采集全分支 → 查找特定回答（可提升）
- **低频但高价值操作**：对比分支、导出内容、分享对话路径（未覆盖）

## 三、改进方案（按优先级排序）

### P0 — 核心体验强化（v2.0 必做）

#### 3.1 国际化 (i18n) 支持

**现状问题**：所有 UI 文本为中文硬编码（"对话树"、"采集图谱"、"调试"等），严重限制了产品在全球 ChatGPT 用户中的传播。

**方案**：
- 提取所有 UI 字符串到语言包对象
- 自动检测浏览器语言，默认 fallback 到英文
- 初期支持：中文 (zh-CN)、英文 (en)
- 语言包在 content.js 内部即可，无需额外文件

**实现复杂度**：低（约 1-2 天）

**预期效果**：消除非中文用户的使用门槛，潜在用户群扩大 10x+

```
// 示例结构
const i18n = {
  'zh-CN': { treeTitle: '对话树', harvest: '采集图谱', debug: '调试' },
  'en':    { treeTitle: 'Chat Tree', harvest: 'Harvest All', debug: 'Debug' }
};
```

---

#### 3.2 分支内容搜索

**需求场景**：用户在一棵有 50+ 节点的对话树中，想找到某个提到 "React hooks" 的回答，但不知道在哪个分支。

**方案**：
- 树面板顶部增加搜索输入框
- 搜索范围：当前对话树中所有节点的消息文本预览
- 匹配节点高亮显示，支持上下箭头切换
- 搜索基于已采集的节点数据，无需额外 DOM 扫描

**实现复杂度**：中（约 2-3 天）

**核心价值**：从"看到树"进化到"在树中找到目标"

---

#### 3.3 键盘快捷键

**需求场景**：Power user 不想每次都移动鼠标点击面板。

**方案**：

| 快捷键 | 功能 |
|--------|------|
| `Alt + T` | 打开/关闭树面板 |
| `Alt + ↑/↓` | 在树节点间上下移动 |
| `Alt + Enter` | 跳转到当前选中的节点 |
| `Alt + H` | 一键采集全分支 |
| `Alt + F` | 聚焦搜索框 |

**实现复杂度**：低（约 1 天）

---

### P1 — 差异化功能（v2.0 重点）

#### 3.4 分支对比视图 (Branch Diff)

**需求场景**：用户重新生成了一段代码解释，想知道两个版本具体哪里不同。

**方案**：
- 在树面板中长按/右键节点进入"对比模式"
- 选择两个同级分支节点（同一问题的不同回答）
- 弹出对比面板，左右并排显示两个回答的文本内容
- 差异部分用颜色高亮（类似 git diff）
- 对比面板支持关闭和最小化

**实现复杂度**：高（约 4-5 天）

**核心价值**：这是当前市场上**没有任何工具提供的功能**，是真正的差异化壁垒

**技术要点**：
- 文本差异算法：使用简单的 LCS 或行级 diff 即可，纯前端实现
- 需要在采集时存储完整消息文本（当前只存预览文本）
- 对比面板可以复用树面板的 DOM 注入模式

---

#### 3.5 分支导出

**需求场景**：用户想保存某个对话分支的完整内容，用于文档、笔记或分享。

**方案**：
- 节点右键菜单增加"导出此分支"选项
- 导出格式：Markdown（默认）、纯文本
- 导出内容：从根节点到选中节点的完整对话路径
- 自动复制到剪贴板 + 可选下载为文件
- 导出格式示例：

```markdown
# ChatGPT 对话导出 — 分支 3/5

**User**: 如何用 React 实现拖拽排序？

**Assistant**: 你可以使用 react-beautiful-dnd 库...

**User**: 有没有不依赖第三方库的方案？

**Assistant**: 可以使用原生 HTML5 Drag and Drop API...
```

**实现复杂度**：中（约 2-3 天）

**核心价值**：让对话树中的内容可以流动到用户的其他工作流中

---

#### 3.6 节点收藏与标注

**需求场景**：一棵大型对话树中，用户想标记 3-4 个关键回答以便快速定位。

**方案**：
- 每个树节点增加星标按钮（默认隐藏，hover 显示）
- 收藏的节点在树中用特殊颜色/图标标识
- 面板顶部增加"仅显示收藏"过滤开关
- 收藏数据持久化到 localStorage

**实现复杂度**：低（约 1-2 天）

---

### P2 — 体验优化（v2.x 后续迭代）

#### 3.7 树节点折叠/展开

**需求场景**：对话树超过 30+ 节点时，面板变得冗长。

**方案**：
- 有子节点的节点增加折叠/展开箭头
- 默认展开当前路径，折叠非活跃分支
- 快捷操作：一键全部折叠/展开

**实现复杂度**：中（约 2 天）

---

#### 3.8 树的缩略图/Minimap

**需求场景**：超大对话树（100+ 节点）时，用户想看到全局结构。

**方案**：
- 面板底部增加一个缩略的 minimap
- 显示整棵树的简化视图（只显示结构线和点）
- 当前视窗区域高亮
- 点击 minimap 可快速跳转到对应区域

**实现复杂度**：高（约 3-4 天）

---

#### 3.9 Options 页面（扩展设置）

**需求场景**：用户想自定义面板位置、快捷键、默认行为等。

**方案**：
- 新增 popup.html + options.html
- 可配置项：
  - 面板默认位置（左/右）
  - 面板默认宽度
  - 快捷键自定义
  - 语言选择
  - 是否自动采集
  - 节点文本预览长度
- 使用 chrome.storage.sync 同步设置

**实现复杂度**：中（约 3 天）

---

#### 3.10 代码架构重构

**现状问题**：全部逻辑在单个 `content.js` 文件中（2369 行），随着功能增加维护成本上升。

**方案**：
- 按职责拆分模块：
  ```
  src/
  ├── core/
  │   ├── controller.js      # 主控制器
  │   ├── tree-model.js      # 树数据模型
  │   ├── dom-scanner.js     # DOM 扫描与消息查找
  │   ├── navigator.js       # 导航与滚动逻辑
  │   └── storage.js         # 缓存与持久化
  ├── ui/
  │   ├── panel.js           # 面板 UI
  │   ├── tree-renderer.js   # 树渲染
  │   ├── diff-viewer.js     # 对比视图
  │   └── theme.js           # 主题检测
  ├── i18n/
  │   ├── zh-CN.js
  │   └── en.js
  └── content.js             # 入口文件
  ```
- 引入简单的构建工具（esbuild）打包为单文件
- 保持零依赖原则，不引入框架

**实现复杂度**：中（约 3-4 天）

**注意**：此项建议在 P0/P1 功能完成后再进行，避免在功能开发期间增加重构负担

## 五、竞品分析

| 维度 | ChatGPT 原生 | ChatGIT v1.1 | ChatGIT v2.0 (规划) |
|------|-------------|-------------|-------------------|
| 分支可见性 | 仅 "1/3" 计数器 | 完整树可视化 | 完整树 + 搜索 + 折叠 |
| 分支导航 | 逐个点击箭头 | 一键跳转任意节点 | + 键盘快捷键 |
| 分支对比 | 不支持 | 不支持 | **并排 diff 对比** |
| 内容导出 | 不支持 | 不支持 | **Markdown 导出** |
| 多语言 | 多语言 | 仅中文 | 中/英 |
| 分支标记 | 不支持 | 不支持 | **收藏标注** |

**市场空白**：目前 Chrome Web Store 上没有提供"对话分支对比"功能的扩展。这是 ChatGIT 建立差异化的最佳切入点。

## 七、风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| ChatGPT DOM 结构变更 | 核心功能失效 | 多层 fallback 选择器（已有）；建立 CI 定期检测 DOM 变化 |
| 单文件代码膨胀 | 开发效率下降 | v2.2 计划模块化重构 |
| 对比功能的文本获取 | 需要采集完整文本而非预览 | 采集时扩展存储内容，渐进式加载 |
| 浏览器扩展审核政策 | 可能被拒绝上架 | 保持零权限申请，不收集用户数据 |
| OpenAI 可能原生实现类似功能 | 产品价值降低 | 持续在对比、导出等高级功能上保持领先 |

## 八、技术实现优先序

建议的开发顺序（考虑依赖关系）：

```
第 1 周：i18n 国际化 + 键盘快捷键
         ↓（低耦合，可并行开发）
第 2 周：分支内容搜索 + 节点收藏标注
         ↓（搜索依赖节点文本数据）
第 3 周：采集增强（存储完整文本）+ 分支导出
         ↓（对比依赖完整文本数据）
第 4 周：分支对比视图
         ↓
第 5 周：测试、打磨、发布 v2.0
```
